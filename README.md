# Custom-Protocol-TM1637

본 프로젝트는 외부 라이브러리에 전혀 의존하지 않고, ATmega328P MCU와 TM1637 모듈 간의 통신 프로토콜을 밑바닥부터 직접 구현한 프로젝트입니다. 단순히 코드를 짜맞추어 동작만 시키는 것에 그치지 않고, 데이터시트의 타이밍 차트, CPU의 명령어 처리 주기(ns), 내부 D-FF 래치 구조 그리고 외부 풀업 저항에 의한 물리적 RC 지연 시간(us)까지 정밀하게 계산하고 분석하여 us 단위에서 완벽한 동기화 제어 시스템을 구축했습니다.

## 📌 Project Overview
<img width="767" height="211" alt="image" src="https://github.com/user-attachments/assets/0ebe5cf4-0721-4ba7-aeb9-a91833758f9e" />
기존에 널리 쓰이는 외부 라이브러리나 아두이노 내장 함수를 일절 배제하고, 하드웨어의 물리적 특성을 완벽히 통제하며 2-Wire 직렬 통신(CLK, DIO)을 소프트웨어 방식(Bit-banging)으로 직접 설계했습니다. 
**1. 데이터시트 기반의 D-Flip Flop 타이밍 제어**
TM1637 내부의 D-FF(래치) 구조를 분석하여 데이터를 핀에 올리고 확정 짓는 타이밍을 정밀하게 제어합니다.
* **Setup Time (CLK Low):** D-FF의 입력문이 닫혀 있는 동안 DIO 전선의 전압을 변경하여, 데이터가 흔들림 없이 안착할 수 있는 안정화 시간을 확보합니다.
* **Latch (CLK Rising Edge):** CLK 전압이 올라가는 셔터 타이밍에 TM1637이 DIO 값을 정확하게 낚아채도록 동기화했습니다.
* **Hold Time (CLK High):** Start/Stop 조건 오작동을 막기 위해 CLK이 High인 구간에서는 DIO 상태를 고정하고 칩이 데이터를 소화할 시간을 제공합니다.

**2. RC 지연(Delay) 및 CPU 클럭 사이클 계산**
`_delay_us()`는 무작정 넣은 대기 시간이 아닙니다. 회로의 물리적 시상수(τ)와 CPU 명령어 처리 속도를 계산하여 도출한 **최적의 안전 마진(5us)**입니다.
* **CPU 처리 속도:** ATmega328P(16MHz)의 1주기는 62.5ns이며, 비트 제어 명령어(`CBI`, `SBI` 등 Read-Modify-Write)는 단 2주기(125ns)만에 실행이 완료됩니다.
* **RC 지연 극복:** 모듈의 10kΩ 풀업 저항과 100pF 커패시터(노이즈 필터)에 의한 시상수(τ)는 1us입니다. 핀을 놓았을 때(High) 5V까지 안정적으로 도달(95% 이상)하려면 최소 3us가 소요됩니다. CPU의 제어 속도가 전압 상승 속도보다 압도적으로 빠르기 때문에 발생하는 신호 꼬임을 방지하고자 5us의 의도적인 지연을 배치했습니다.

**3. Open-Drain 방식을 통한 과전류(Short Circuit) 보호**
MCU와 TM1637이 DIO 선의 제어권을 주고받을 때(특히 9번째 클럭 ACK 수신 시) 양측이 서로 다른 전압(5V와 0V)을 뿜어내어 칩이 타버리는 버스 충돌 현상을 원천 차단했습니다.
* **Low 출력:** `DDR`을 출력 모드로 설정하여 내부 GND 스위치를 닫고 전압을 0V로 강하게 끌어내립니다 (Sink).
* **High 출력:** `DDR`을 입력 모드로 전환하여 MCU는 선에서 완전히 손을 뗍니다(High-Z). 이후 외부 풀업 저항의 힘만으로 자연스럽게 5V가 채워지도록 유도하여 안전성을 극대화했습니다.

## 🔬 TM1637 데이터시트 기반 회로도 분석
<img width="758" height="477" alt="image" src="https://github.com/user-attachments/assets/5d019174-4765-4841-8d5f-f97cfbbf379a" />

TM1637 모듈의 통신 선(CLK, DIO)에는 외부 풀업 저항(10kΩ)과 노이즈 필터용 커패시터(100pF)가 달려 있습니다. 따라서 MCU 핀을 **입력(Input) 모드**로 전환하면, 외부 풀업 저항을 통해 해당 핀의 전압이 5V로 자연스럽게 올라갑니다.

하지만 커패시터와 저항으로 인한 **RC 지연(RC Delay)** 때문에 전압이 즉각적으로 변하지 않고 곡선을 그리며 서서히 상승합니다. 이를 시상수(τ)로 계산해 본 결과, 논리적인 High 상태에 안정적으로 도달하려면 최소 3us 이상의 대기 시간이 필요하다는 것을 확인했습니다.

또한, 이 외부 풀업 저항 환경을 활용하여 **오픈 드레인 제어 방식**을 적용했습니다. 덕분에 MCU가 5V를 강제로 출력하고 있을 때 TM1637이 0V로 끌어내리려 하면서 발생하는 **버스 충돌 및 과전류 파손 위험**을 완벽하게 막을 수 있었습니다.

이처럼 데이터시트와 회로도를 직접 분석함으로써, 단순한 감이 아니라 하드웨어적 근거를 바탕으로 명령어 사이사이의 지연 시간과 전압 안정화 타이밍을 설계할 수 있었습니다.

## 💡 프로젝트 후기 및 느낀 점

이번 프로젝트는 단순히 디스플레이에 숫자를 띄우는 '작동하는 코드'를 짜는 것을 넘어, **소프트웨어와 하드웨어의 물리적 경계가 만나는 지점**을 깊이 이해하게 된 뜻깊은 경험이었습니다. 남들이 만들어둔 편한 라이브러리를 사용했다면 5분 만에 끝났을 작업이지만, 밑바닥부터 통신 프로토콜을 us 시간 단위에서 구현하며 다음과 같은 값진 깨달음을 얻었습니다.

**1. "정확한" 딜레이 계산**
이전에는 하드웨어 통신 중 에러가 나면 무작정 딜레이 시간을 늘려가며 '감'으로 코딩을 하곤 했습니다. 하지만 이번에는 달랐습니다. 
모듈에 달린 풀업 저항(10kΩ)과 커패시터(100pF)의 시상수(τ)를 직접 계산하여 전압이 5V까지 차오르는 물리적인 시간(3us)을 구하고, 이를 CPU의 비트 제어 명령어 처리 속도(125ns)와 비교했습니다. 그 결과 도출해 낸 **'안전 마진을 포함한 5us의 딜레이'**를 코드에 넣었을 때의 쾌감은 이루 말할 수 없었습니다. 내가 작성한 코드 한 줄 한 줄에 명확한 수학적, 물리적 근거가 생겼다는 것이 가장 큰 성과입니다.

**2. 데이터시트 분석**
TM1637의 타이밍 다이어그램과 D-Flip Flop 논리 회로 구조를 분석한 끝에 완벽한 동기화 타이밍을 찾아낼 수 있었습니다. 
`CLK`이 Low일 때 데이터를 안전하게 준비(Setup)하고, High로 올라가는 Rising Edge에서 데이터를 낚아채는(Latch) 과정을 직접 구현해 보면서, 데이터시트를 소프트웨어 로직으로 번역해 내는 능력이 크게 향상되었습니다.

**3. "소프트웨어의 하드웨어를 망치는 실수" (Open-Drain의 중요성)**
MCU와 TM1637이 9번째 클럭(ACK)에서 제어권을 주고받을 때, 만약 양쪽이 서로 다른 전압(5V와 0V)을 강제로 뿜어내면 쇼트가 발생해 과전류로 칩이 탈 수도 있다는 물리적 위험성을 깨달았습니다. 
이를 예방하기 위해 MCU 측에서 강제로 5V를 출력하지 않고 핀을 놓아주는 **오픈 드레인 제어 방식**을 고안 및 적용했습니다. 이를 통해 임베디드 개발자는 단순히 알고리즘을 잘 짜는 것을 넘어, 하드웨어의 전기적 안전성까지 책임지고 방어하는 코드를 작성해야 한다는 것을 배웠습니다.

**🚀 결론**
이번 프로젝트를 통해 **단순히 API를 가져다 쓰는 개발자에서 벗어나, 하드웨어의 동작 원리를 로우 레벨(Low-level)까지 통제하고 제어할 줄 아는 엔지니어**로 한 걸음 성장할 수 있었습니다. 앞으로 어떤 칩과 센서를 마주하더라도 데이터시트를 분석해 나만의 드라이버를 밑바닥부터 작성할 수 있다는 강한 자신감을 얻었습니다.
